# Session List Batch Ops Design

日期：2026-02-27

## 背景與目標

目前「課堂列表」與「批次指派老師精靈」功能分散，管理員需要在不同入口切換。  
本次目標是把主要操作收斂到課堂列表，降低操作成本，同時保留衝突安全機制。

## 已確認決策

1. 批次操作入口統一在「課堂列表」。
2. 保留跨頁全選（依目前篩選條件）。
3. 日期調整僅支援單堂修改，不做批次改日期。
4. 批次改時間、批次停課都採固定傻瓜流程：
- 先預覽（dryRun）
- 顯示可套用與衝突數量
- 只允許「套用可更新項目」與「取消」
5. 不提供 strict/force/skip-conflicts 給使用者選擇，系統固定略過衝突。
6. 必須提供衝突處理流程提示（可操作、非純錯誤訊息）。

## UX 流程（傻瓜版）

### 批次改時間

1. 管理員在課堂列表勾選課堂或使用「依篩選條件跨頁全選」。
2. 點擊「批次改時間」，輸入新開始/結束時間。
3. 系統自動預覽，顯示：
- 可更新 X 堂
- 衝突 Y 堂（不會修改）
4. 操作者可選：
- 套用可更新項目（主按鈕）
- 查看衝突清單（次按鈕）
- 取消
5. 套用後回到課堂列表並顯示成功/略過摘要。

### 批次停課

1. 入口與預覽流程同批次改時間。
2. 預覽顯示可停課與不可停課數量（例如已完成課堂不可停課）。
3. 僅套用可停課項目。

### 單堂改日期

保留在單堂操作（既有調課流程），不納入批次工具。

## 衝突處理提示（固定區塊）

在預覽結果上方顯示：

- 標題：`偵測到衝突，已自動略過`
- 摘要：`本次可更新 {okCount} 堂，{conflictCount} 堂需先處理。`
- 三步提示：
  1. 先套用可更新項目
  2. 切到只看衝突逐筆修正
  3. 重新預覽後再套用

衝突清單原因分類：

1. 老師撞堂
2. 同班時段重疊
3. 狀態不可修改
4. 時間格式/範圍不合法

## 系統架構與責任

### Frontend

1. `session-list-dialog` 成為主要批次入口容器。
2. 新增批次工具列（批次改時間、批次停課）。
3. 新增「跨頁全選（依篩選）」狀態管理與可視化摘要。
4. 新增預覽結果區塊與衝突提示區塊。
5. 保留單堂操作入口（改日期/改時間/停課）。

### Backend

1. 延用 `dryRun -> apply` 兩段式模型。
2. 新增/調整批次 API（建議）：
- `PATCH /api/classes/:id/sessions/batch-update-time`
- `PATCH /api/classes/:id/sessions/batch-cancel`
3. API 固定回傳：
- `processableIds`
- `conflicts`
- `summary`（updated/skipped）
- `dryRun`
4. 套用時僅處理 `processableIds`。
5. 全部操作寫入 audit 與異動紀錄。

## 資料流

1. 前端送出篩選條件 + 目標集合策略（勾選或跨頁全選）+ 操作參數。
2. 後端計算實際目標課堂集合。
3. 後端先做衝突檢查並回傳預覽結果。
4. 前端顯示可套用/衝突摘要與處理提示。
5. 使用者確認套用後，前端送 apply 請求（帶 preview token 或 processableIds）。
6. 後端只更新可處理項目，回傳最終統計。

## 錯誤處理策略

1. 不將衝突視為致命錯誤，而是可預期結果。
2. 致命錯誤僅限：
- 權限不足
- 請求格式錯誤
- 資料已不存在
- 伺服器失敗
3. 致命錯誤以 toast + 區塊錯誤提示雙層呈現。
4. 預覽與套用結果皆保留明確數字，避免「看起來成功但其實略過很多」。

## 測試策略

### Frontend

1. 跨頁全選狀態正確（篩選改變時重算/失效處理）。
2. 預覽結果 UI（有衝突/無衝突）正確渲染。
3. 主流程僅出現「套用可更新項目」與「取消」。
4. 手機版分頁與衝突清單不破版。

### Backend

1. 批次改時間：
- 正常可改
- 老師衝突被略過
- 同班衝突被略過
- 非法時間拒絕
2. 批次停課：
- scheduled 可停課
- completed/cancelled 略過
3. dryRun 與 apply 的統計一致性。
4. audit 與異動紀錄完整。

## 非目標（本階段不做）

1. 不做批次改日期。
2. 不提供 strict/force 高階模式。
3. 不在本階段重做全域課堂管理頁，只在既有課堂列表內演進。
